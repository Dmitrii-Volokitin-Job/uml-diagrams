Introduction

What is UML?

Unified Modeling Language — a standardized visual language for software architecture. It is not a methodology or programming language, but a notation system.

The Core Idea

Before UML, every team invented its own notation. The "Three Amigos" — Grady Booch, Ivar Jacobson, James Rumbaugh — unified three popular OO methods at Rational Software, producing UML 1.0 in 1997. The OMG (Object Management Group) standardized it. Today UML 2.5.1 (2017) is the current version and the one all professional tools implement.

UML gives teams a shared vocabulary: a developer in Berlin and an architect in Tokyo can exchange a diagram without ambiguity, because every symbol has a defined, specification-level meaning.

The 4+1 View Model

Philippe Kruchten's 4+1 model (1995) explains why we need multiple diagram types. One diagram can never capture the full system:

  • Logical View — object model, classes (→ Class Diagram, Object Diagram)
  • Process View — concurrency, threads, processes (→ Sequence, Activity, State Diagrams)
  • Development View — software module organization (→ Package, Component Diagrams)
  • Physical View — hardware mapping (→ Deployment Diagram)
  • +1: Scenarios — use cases that tie the 4 views together (→ Use Case Diagram)

Diagrams are views, not the model

In formal UML, a model is the underlying semantic data. Diagrams are views of it. In practice, most teams skip the formal model and treat diagrams as the artefact itself — which is fine. What matters is communication clarity, not spec compliance.

History & Quick Reference

Know the versions to avoid outdated tutorials. The quick-reference table helps you pick the right diagram fast.

Timeline
  • 1994–96 — Booch Method, OMT (Rumbaugh), OOSE (Jacobson) merged at Rational Software.
  • 1997 — UML 1.0 → OMG standard. Class, Sequence, Use Case dominate.
  • 2005 — UML 2.0: major revision. Adds Timing Diagram, Interaction Overview, Composite Structure. Activity Diagrams completely redesigned (now based on Petri nets). "Collaboration Diagram" renamed "Communication Diagram".
  • 2017 — UML 2.5.1: current. Simplified specification language, no new diagram types. This is what Sparx EA, IntelliJ, PlantUML implement.

UML 1.x vs 2.x gotchas

Collaboration Diagram = UML 1.x name for Communication Diagram.
Activity Diagram was completely redesigned in 2.0 — older tutorials show incorrect notation (no fork/join bars).
Timing & Interaction Overview diagrams did not exist in UML 1.x.

How to pick the right diagram
  • Start with your audience — a Use Case diagram talks to product owners; a Deployment diagram talks to ops. Pick the diagram that serves the reader first.
  • One concern per diagram — don't mix structural and behavioral concerns. Use separate diagrams and cross-reference them with «ref» or hyperlinks.
  • Prefer the simplest diagram that answers the question — if an Activity Diagram answers it, don't reach for an Interaction Overview Diagram.
  • Draw the target state, not the current mess — diagrams are design tools, not archaeology. Only document "as-is" when it is a shared problem statement that everyone needs to acknowledge.

Class Diagram

The most used diagram in UML. Shows classes, interfaces, attributes, methods, and all their relationships. The static backbone of object-oriented design.

Theory

A class diagram is a static structural view of a system. It models the concepts in the problem domain and the relationships among them. Every class is a template (blueprint); the Class Diagram is not about runtime — it is about design-time structure.

A class has three compartments: name (top, bold), attributes (middle), and operations (bottom). Visibility prefixes: + public, - private, # protected, ~ package. Type follows the name after a colon: - balance : double.

Class diagrams can represent abstract classes (name in italics or {abstract}), interfaces («interface» stereotype), and enumerations («enumeration»).

Class anatomy + all six relationship types
Use it when
  • Designing entities before writing code — class diagrams are the best tool to align the team on the domain model.
  • Reverse-engineering existing code with tools like IntelliJ's "Diagram" feature to understand legacy systems.
  • Code review documentation — attaching a class diagram to a PR makes architectural intent obvious.
  • Database design — a class diagram often precedes the ER diagram; they map closely.
Don't use it when
  • Showing behaviour or order — a class diagram is static. Use Sequence or Activity instead.
  • You want to model every field — showing private helper fields and internal constants clutters the diagram with noise. Focus on the design-level concept.
  • The team hasn't agreed on requirements — don't design classes before understanding the domain. Do Use Case first.
  • Data transfer objects — DTOs are boring flat structures; drawing them wastes time.
Best Practices
  • Top-down: Start with 5–8 core domain classes. Expand later. Never try to diagram the whole codebase at once.
  • Noun = class, verb = operation, adjective = attribute: If you're struggling to name something, check whether it's actually the wrong element type.
  • Composition over aggregation: If you're unsure which to use, ask "can B exist if A is deleted?" If not, use composition (filled diamond).
  • Avoid bidirectional associations unless truly needed — they couple classes and make dependency direction unclear.
  • Name every association with a verb phrase (e.g., placedBy, contains) to communicate intent.
  • Separate interface from implementation — show interfaces as «interface» with a dashed realization arrow to implementing classes.
Common Mistakes
  • Showing every field and method — a class with 30 attributes is noise, not documentation. Show design-level fields only.
  • Overusing inheritance — inheritance is the strongest coupling in OOP. Favour composition. If you see 4+ levels of inheritance, stop and redesign.
  • Forgetting visibility prefixes — + - # ~ encode the interface contract. Omitting them makes the diagram useless for encapsulation design.
  • Missing multiplicity — an association without 1, *, 0..1 etc. is ambiguous. Is Customer → Order one-to-one or one-to-many?
  • Putting every class on one diagram — split by domain or layer. A diagram that can't fit on an A4 page needs to be split into multiple focused diagrams.

Object Diagram

A snapshot of a system at one instant — shows concrete instances of classes with real values. The Class Diagram is the blueprint; the Object Diagram is the photograph.

Theory

An Object Diagram is a specific instance view of a class diagram at one point in time. It uses the same notation as class diagrams but shows objects (instances) rather than classes. Object names are underlined and written as instanceName : ClassName. Attribute values are actual values, not types.

Links (object-level connections) correspond to associations in the class diagram. Object diagrams are particularly useful for illustrating complex data structures: trees, graphs, linked lists, or recursive relationships that are hard to understand from the class diagram alone.

Object diagram — an order placed by Alice
Use it when
  • Explaining recursive structures — a linked list or tree is much clearer as a concrete object diagram than an abstract class diagram.
  • Validating your class design — instantiate a specific scenario mentally; if it's awkward to draw, your class design may be wrong.
  • Onboarding new developers — a concrete snapshot with real values is far easier to understand than abstract boxes.
  • Debugging complex state — drawing the actual object graph in production helps isolate bugs.
Don't use it when
  • As living documentation — Object Diagrams become stale the moment code changes. Use them to explain, not to document.
  • Replacing unit tests — a diagram is not executable. Write tests to verify object state.
  • Large systems — more than 8–10 objects and the diagram becomes a tangled web. Slice to a relevant subset.
Best Practices
  • Pick ONE scenario: an object diagram should illustrate exactly one meaningful situation (e.g., "Gold customer with a pending multi-item order").
  • Use realistic values — "alice@example.com" communicates more than "email1". Use domain-appropriate data, not placeholder strings.
  • Pair with its class diagram — always show both; the object diagram only makes sense relative to the class diagram it instantiates.
  • Name the scenario in the diagram title — "Object Diagram: Gold customer placing multi-item order (Jan 2024)" makes the purpose and limitations clear.
  • Use to validate class design — if you can't easily draw a concrete object diagram from your class diagram, the class design has a flaw. The object diagram is a design test.
  • Show the interesting connections, not all objects — if a scenario has 50 objects, show the 4–5 directly relevant to the concept being illustrated. Add a note "only relevant objects shown".
Common Mistakes
  • Using placeholder values — "name1", "val1" defeats the purpose. The diagram exists to show one concrete, believable scenario. Use data that tells a story.
  • Too many objects — if you show 15 objects you are drawing a database schema, not illustrating a design concept. Keep it to the 3–6 objects that demonstrate the specific scenario.
  • Unlabelled links — every link between objects should carry the association name or role it instantiates from the class diagram. An unmarked line says nothing about WHY these objects are connected.
  • No matching class diagram — an object diagram in isolation is unreadable. Readers need to see what type each instance comes from. Always pair the two.

Component Diagram

Shows how a system is divided into modular, deployable components and how they connect through provided/required interfaces. Think: microservices, JARs, NPM packages, REST APIs.

Theory

A component is a self-contained, replaceable unit that encapsulates its implementation and exposes a set of interfaces. This is the key difference from a class: a component may contain many classes, and it is typically a deployable artifact (.jar, .dll, Docker image, microservice).

Provided interface (lollipop ○—): what the component offers to the outside world.
Required interface (socket ⊂○): what the component needs from another component.
A ball-and-socket connector links a provided interface of one component to a required interface of another — representing a dependency that is fulfilled.

UML also distinguishes between interface components (pure API contracts) and implementation components (concrete implementations). This maps directly to Java interfaces + Spring beans, or TypeScript interfaces + service classes.

Microservice architecture — component view
Use it when
  • Architecture review with management or new team members — one page showing all services and how they talk.
  • Planning microservices — decide service boundaries and API contracts before coding.
  • Dependency audit — find which components depend on which third-party libraries.
  • Integration testing planning — the component diagram shows which interfaces need test doubles.
Don't use it when
  • Single-module monolith — if there's only one deployable artifact, a package diagram gives better value.
  • Showing runtime message flow — that's a Sequence or Activity Diagram's job.
  • Class-level design — use a Class Diagram to show internals.
Best Practices
  • One component = one team boundary — if two components are always changed together, merge them.
  • Show both provided AND required interfaces — makes contractual obligations visible at a glance.
  • Group components by tier (frontend / backend / data) using dashed boundary boxes.
  • Label connections with protocols (HTTPS, AMQP, JDBC) so ops teams can understand the diagram too.
Common Mistakes
  • Drawing classes as components — a component is a deployable or replaceable unit, not a class. If you have 20 "components", you are probably drawing a class diagram with wrong notation.
  • Omitting interfaces — showing components with plain arrows and no interface labels hides the contractual obligation. The lollipop (provided) and socket (required) notation exist for a reason: they make the contract explicit.
  • Mixing runtime and build-time views — "Docker container" and "Java class" live at different abstraction levels. Keep one abstraction level per diagram.
  • No system boundary — without a boundary box, readers cannot tell which components are yours and which are external third-party dependencies.

Deployment Diagram

Shows the physical and virtual infrastructure: nodes (servers, devices), execution environments (Docker, JVM), artifacts (JARs, WAR files) and how they are connected.

Theory

A node is a computational resource — either a device (physical hardware: server, phone, router) or an execution environment (software runtime: Docker container, JVM, web browser, Kubernetes pod). Nodes are drawn as 3D boxes.

An artifact is a physical file deployed on a node: api.jar, app.war, a Docker image. Artifacts live inside nodes or execution environments.

Communication paths are solid lines between nodes, labelled with protocols (TCP/IP, HTTPS, JDBC, AMQP). They represent physical network connections.

Nodes can be nested: a physical server contains Docker containers, which contain JVM instances, which contain deployed artifacts. This nesting accurately represents modern cloud infrastructure.

AWS cloud deployment topology
Use it when
  • Planning cloud or on-prem infrastructure before provisioning resources.
  • DevOps / ops handoff — show SREs exactly what runs where.
  • Security reviews — network paths show trust boundaries and data flows for threat modelling.
  • Disaster recovery planning — shows which nodes are single points of failure.
Don't use it when
  • Modelling application logic — that's for Class, Sequence, or Component diagrams.
  • Early design phase — infrastructure decisions should follow architecture decisions, not precede them.
  • Very simple deployments — one server + one DB doesn't need a diagram; a README paragraph suffices.
Best Practices
  • Annotate with quantities — "EC2 × 2–10 (autoscale)" conveys capacity planning intent.
  • Label network zones — mark VPCs, subnets, DMZs, private/public to show security boundaries.
  • Include version numbers — "PostgreSQL 16.1" ages a diagram but also helps audits.
  • Show redundancy — if a service runs on 2 nodes, draw both; single points of failure become visible.
Common Mistakes
  • Mixing logical and physical views — components (what runs) belong in a Component Diagram; deployment diagrams show WHERE things run. Don't put class names on deployment nodes.
  • Unlabelled communication paths — a line between EC2 and RDS says nothing. Label it with protocol (JDBC), port (5432), and encryption (SSL/TLS). Auditors and ops engineers need this information.
  • Everything on one node — if every artifact runs on the same node, the diagram carries zero deployment information. Use a Component Diagram instead and note "single-node deployment".
  • Missing security zone boundaries — a diagram without public/private subnet, VPC, or DMZ boundaries cannot be used for security reviews. Show where trust boundaries cross.
PlantUML example
plantuml
@startuml node "AWS EC2" { node "Docker" { artifact "api-service.jar" } } database "AWS RDS" { artifact "PostgreSQL" } "AWS EC2" ..> "AWS RDS" : JDBC @enduml

Package Diagram

Organizes model elements into named namespaces (packages) and shows dependencies between them. The primary tool for enforcing layered architecture and detecting circular dependencies.

Theory

A package maps directly to Java packages, C# namespaces, Python modules, or any other modular grouping. It groups related classes, interfaces, and other packages.

Three types of dependency between packages:

  • «import» — public import; imported elements are visible to clients of the importing package.
  • «access» — private import; elements are visible only inside the importing package.
  • «merge» — merges the content of one package into another (advanced; used in UML metamodel itself).

The most important architectural principle expressible in a Package Diagram: the Dependency Rule — dependencies should point inward, toward more stable layers. Domain core has zero outgoing dependencies. Infrastructure depends on domain, not the other way around.

Clean Architecture layers
Use it when
  • Establishing architecture at the start of a project — agree on packages before creating classes.
  • Circular dependency detection — if drawing the arrows creates a cycle, there's an architectural violation.
  • Architecture fitness functions — many tools (ArchUnit, Structure101) can check that actual package dependencies match the diagram.
Don't use it when
  • Showing every package in a large codebase — a diagram with 50 packages is unreadable. Focus on the key layers.
  • No modular structure exists yet — if the team puts everything in one package, draw the target state, not the current mess.
Best Practices
  • Draw the target state — use the diagram to communicate where you want to get to, then enforce it with build tools (ArchUnit, Checkstyle).
  • One arrow direction only — each layer should only depend on layers "below" it (more abstract/stable). Arrows going "up" are architectural violations.
  • Named dependencies — label with «import» or «use» to show the relationship nature.
  • Enforce with build tools — a beautiful package diagram that nobody checks against the code is history, not architecture. Wire it to ArchUnit, Checkstyle, or Dependency-Check in CI.
Common Mistakes
  • Diagramming the current mess — if your codebase violates Clean Architecture, don't diagram the violations as-is. Draw the target state and use it to drive refactoring with a clear migration path.
  • Circular dependencies — A depends on B which depends on A. The diagram reveals this immediately. The fix is an interface, an event, or a domain service. Never leave cycles in a package diagram — they signal a real design problem.
  • Too granular — packaging every class into its own "package" produces noise. A package should represent a cohesive module with a clear responsibility, not a single file.
  • Missing import direction labels — an unlabelled arrow between two packages is ambiguous. Always specify «import» (public re-export) vs «use» (internal usage) vs «merge» (additive).
PlantUML example
plantuml
@startuml package "presentation" #DDEEFF { } package "application" #DDFFD4 { } package "domain" #EEE4FF { } package "infrastructure" #FFF4CC { } "presentation" ..> "application" : «use» "application" ..> "domain" : «use» "infrastructure" ..> "domain" : «use» @enduml

Composite Structure Diagram

Shows the internal structure of a class or component — its named parts, ports, and internal connectors. Introduced in UML 2.0 for embedded systems and design pattern documentation.

Theory

While a Class Diagram shows relationships between classes, a Composite Structure Diagram shows the internal decomposition of a single classifier. Key concepts:

  • Part: a named slot inside the class (role it plays). Written as partName : ClassName. Parts have multiplicity.
  • Port: an interaction point on the boundary of the class (shown as a small square on the edge). A port has a type (interface or class).
  • Connector: a link between parts (or between a port and a part) showing how they interact internally.
  • Delegation connector: links a port on the boundary to an internal part — showing that external requests arriving at the port are handled by that part.

This diagram is especially valuable for documenting design patterns (the internal wiring of a Facade, Mediator, or Decorator), and for embedded systems where one MCU has multiple hardware sub-components.

OrderProcessor — internal structure with ports
Use it when
  • Documenting design patterns — Composite Structure shows the wiring of a Facade, Mediator, or Bridge pattern much more clearly than a class diagram.
  • Embedded systems — one device (MCU) with multiple sub-components (sensor, actuator, communication module).
  • Framework internals — documenting how a framework component delegates responsibilities to pluggable sub-parts.
Don't use it when
  • Simple classes with a few fields — overkill. Use a class diagram compartment.
  • Showing inter-class relationships — that's a class diagram's job. Composite Structure is for intra-class structure only.
  • Most business application teams — this diagram is rarely needed outside embedded or framework development.
Best Practices
  • Draw it only when the internal structure is the design decision — if the wiring is obvious from the class diagram, skip it.
  • Use typed ports — name ports with the interface they expose (orderIn : IOrderProcessor). Anonymous ports carry no contractual information.
  • Show delegation connectors explicitly — they make the boundary between "external contract" and "internal implementation" visible and testable.
  • Limit to 3–5 parts — more than that signals the class itself needs to be split into smaller components.
  • Pair with the Class Diagram — each Part in a Composite Structure Diagram should correspond to a private field or association in the Class Diagram.
Common Mistakes
  • Confusing parts with classes — a Part is an instance slot (a role inside this classifier), not a class. You can have two Parts of the same type playing different roles inside one composite, e.g., two : Timer parts with roles heartbeatTimer and requestTimer.
  • Anonymous ports — a port without an interface type is a design smell. Every port must name the interface it exposes (orderIn : IOrderProcessor). Untyped ports cannot be validated or mocked in tests.
  • Omitting delegation connectors — drawing a boundary port without showing what internal part it delegates to leaves the diagram incomplete. The delegation connector IS the non-obvious design decision this diagram exists to document.
  • Over-diagramming obvious wiring — if the internal structure is already clear from the Class Diagram, a Composite Structure Diagram adds noise. Use it only when ports, parts, and delegation are the insight that cannot be read from a class diagram alone.

Profile Diagram

The mechanism for extending UML itself. Defines stereotypes, tagged values, and constraints for a specific domain or platform (Spring, Java EE, security models). Rarely needed in application development.

Theory

UML is generic by design. A Profile is an extension package that adds domain-specific meaning without changing the UML metamodel. It does this through:

  • Stereotype: extends a UML metaclass (Class, Component, Association, etc.) by adding new semantics. Written as «StereotypeName» on model elements. Example: «Repository» extends Class to mean "a Spring data repository".
  • Tagged values: additional attributes on stereotypes. Example: «Repository» might have a tagged value {entity = Order}.
  • Constraints: OCL (Object Constraint Language) rules that model elements with the stereotype must satisfy.

When your UML tool knows the Spring profile, it automatically validates that any class marked «Service» doesn't extend another «Repository». Profiles are the foundation of Model-Driven Architecture (MDA) — generating code from annotated models.

Spring Framework profile (simplified)
Use it when
  • Building a framework that other teams model against (a custom domain-specific extension of UML).
  • Model-Driven Architecture — generating code from annotated models using transformations.
  • Enterprise-wide UML standards — define stereotypes your whole organisation uses.
Don't use it when
  • Application development — you use profiles from your tool's Spring/JEE plugin; you don't write them.
  • Quick documentation — a code comment or README is faster and more accessible.
  • Teams new to UML — master the other 13 diagram types first. Profile Diagrams are specialist territory.
Best Practices
  • One profile per technology layer — a "Spring Backend Profile" and a "JPA Persistence Profile" are more reusable than one giant "App Profile".
  • Always extend an existing UML metaclass — your stereotype extends Class, Component, Association, etc. Never extend another custom stereotype in the same profile (creates fragile chains).
  • Name stereotypes with target-tech naming — use @Service not ServiceLayer. This makes the mapping to implementation annotations obvious.
  • Write at least one OCL constraint per stereotype — even simple ones prevent misuse. Example for @Repository: self.ownedAttribute → select(a | a.type.name = 'Long') → notEmpty() (must have an ID field).
  • Use profiles with a modelling tool that supports them — raw diagram tools (draw.io) don't validate profiles. Enterprise Architect, MagicDraw, or Visual Paradigm do.
Common Mistakes
  • Using comments instead of stereotypes — writing // Spring controller in a class gives no structural semantics. A stereotype lets tools validate, generate code, and transform models.
  • A stereotype for every annotation — adding a stereotype for every framework annotation bloats the profile. Focus on stereotypes that carry meaningful tagged values or OCL constraints that enforce real design rules.
  • Extending another custom stereotype — always extend a standard UML metaclass (Class, Component, Association). Chaining custom stereotypes creates fragile dependency chains within the profile itself and breaks tool validation.
  • No OCL constraints — a stereotype without constraints is just a label. OCL constraints are what allow tools to verify that the stereotype is applied correctly and consistently across the model.

Use Case Diagram

Captures functional requirements from the user's perspective. Who does what with the system? A communication tool between developers and non-technical stakeholders.

Theory

A Use Case Diagram shows actors (external entities that interact with the system) and use cases (system functionalities valuable to actors). The system boundary rectangle separates the system from the outside world.

Actors can be human users, external systems, or time-triggered events. Each actor communicates with the system by participating in use cases.

Two special relationships between use cases:

  • «include»: the base use case always delegates to the included one. (Placing an Order always includes Authenticate.) Arrow points from base → included.
  • «extend»: the extending use case is optional — it adds behaviour to the base use case at a defined extension point, only when a condition is met. Arrow points from extension → base.

Use cases also support generalization (both actor and use case level). A "Manager" actor generalizes "Employee" and inherits all their use cases.

Cockburn's use case levels: Summary (sea: order a product), User goal (kite: register account), Subfunction (clam: validate email format). Only User goal level belongs on a use case diagram — not subfunctions.

E-commerce system use cases
«include» vs «extend» — the #1 confusion in UML

«include»: the base case ALWAYS executes the included case — it's a mandatory call. Think of it like a method call. Arrow points FROM base TO included.
«extend»: the extension is OPTIONAL, triggered by a condition at an extension point. Arrow points FROM extension TO base.
Memory trick: include = always, extend = sometimes.

Use it when
  • Requirements phase — the only diagram non-technical stakeholders can read fluently.
  • Scope definition — the system boundary rectangle is a powerful tool to say "this is in scope, this is not".
  • Sprint planning — each user-goal use case is roughly one epic.
Don't use it when
  • Describing UI flows — use case diagrams describe goals, not screens or click sequences.
  • Replacing a use case spec — the diagram is a map; you still need written use case descriptions for each oval.
  • Technical design — use case diagrams say WHAT, not HOW. Never put classes or data flows on them.
Best Practices
  • User goal level only — each use case should be achievable in one sitting by the actor and deliver observable value.
  • Start with actors — list every external entity first, then ask "what does each one need to do?"
  • Keep the diagram simple — 5–10 use cases per diagram. Use multiple diagrams for large systems.
  • Always draw a system boundary — it forces an explicit scope decision.
  • Avoid functional decomposition — don't turn a use case into a flowchart. That's an Activity Diagram.
Common Mistakes
  • Functional decomposition — "Login → Validate Email → Check Password → Return JWT" is a flowchart of steps, not a set of user goals. Each oval must represent one complete, observable goal that an actor cares about.
  • Person as actor instead of role — "Alice the admin" is wrong; "Administrator" is right. Multiple people play the same role, and the same person can play multiple roles depending on context.
  • Overusing «include» — not every shared step needs an «include» relationship. Use include only when a sub-flow is genuinely reused across 3+ use cases or needs to be documented separately. Overuse creates a decomposition, not a use case model.
  • No system boundary — a use case diagram without the boundary rectangle is just floating ovals with no scope statement. The boundary IS the most important design decision on the diagram.
  • Technical steps as use cases — "Validate JWT Token", "Run Database Query" are system internals, not user goals. If a non-technical stakeholder wouldn't recognise the goal as meaningful to them, it doesn't belong as a use case.

Activity Diagram

Models workflows and algorithms as a formal flowchart. Based on Petri net theory (UML 2.0+). Supports sequential flow, parallel execution, decisions, signals, and swimlanes.

Theory

Activity Diagrams are the UML equivalent of flowcharts, but with formal semantics borrowed from Petri nets. Tokens flow through nodes (activities, decisions, forks, joins) along edges. Multiple tokens can exist simultaneously, enabling modelling of true concurrency.

Key nodes:

  • Initial node (●): where the flow begins. Only one token enters.
  • Action node: a unit of work (rounded rectangle). Can have pre-/post-conditions.
  • Decision node (◇): guard conditions on outgoing edges select ONE path.
  • Fork (thick bar): ONE incoming edge, MULTIPLE outgoing — spawns parallel flows (multiple tokens).
  • Join (thick bar): MULTIPLE incoming edges, ONE outgoing — synchronizes all parallel flows (waits for all tokens).
  • Merge (◇): multiple inputs, one output — no synchronization (first token through wins).
  • Activity final (◎): terminates the entire activity when any token reaches it. Flow final (⊗): terminates only that specific flow token — important distinction.
  • Signal send/receive: activity nodes shaped like a flag (send) or pentagon (receive) for async communication.
  • Swimlanes (partitions): divide the diagram vertically or horizontally by actor/system — show who is responsible for each action.
Checkout process with swimlanes, fork/join, and signals
Use it when
  • Business process modelling — activity diagrams with swimlanes are the UML equivalent of BPMN for quick collaboration.
  • Documenting algorithms — especially when a flowchart needs to show parallel branches.
  • Clarifying complex workflows before implementation — e.g., order fulfilment, approval chains.
  • Multiple actors / systems involved — swimlanes make responsibility instantly visible.
Don't use it when
  • Simple linear flows — a numbered list is faster to read than a diagram with no branches.
  • Showing message details — use a Sequence Diagram when you care about parameter names, return values, and call order within a scenario.
  • Object lifecycle — use a State Machine Diagram for one object's lifecycle states.
Best Practices
  • Name actions as verb + noun: "Validate Order", not "Validation" or "Order".
  • Always close parallel branches — every fork must have a corresponding join. Missing joins are logic bugs.
  • Use flow final (⊗) not activity final (◎) when terminating only one branch in a parallel scenario — otherwise you kill the whole activity.
  • Limit nesting — deeply nested activities are hard to read. Extract sub-activities using a "call action" node that references another diagram.
Common Mistakes
  • Decision vs merge node confusion — a diamond with one input and multiple outputs is a decision. A diamond with multiple inputs and one output is a merge. Mixing them breaks the formal token semantics.
  • Unclosed forks — every fork bar (parallel split ─┬─) must have a corresponding join bar (─┴─). An open fork means the activity never terminates in the formal sense — a logic bug, not just a visual issue.
  • Flow final vs activity final — ⊗ (flow final) ends ONE token on one branch; ◎ (activity final) ends the ENTIRE activity. Using ◎ inside a parallel branch prematurely kills all concurrent threads. This is one of the most common logic errors in Activity Diagrams.
  • Technical components in swimlanes — swimlanes represent human roles or organisational responsibility (Customer, Order Service team). Putting "AWS Lambda" or "Database" in a swimlane conflates responsibility with infrastructure — use a Sequence Diagram for that.

State Machine Diagram

Models the lifecycle of an object — every state it can be in and the transitions between states triggered by events. Also called Statechart. Based on David Harel's statecharts (1987).

Theory

A state machine models how one reactive object changes state in response to events. Key elements:

  • State: a condition in which an object waits for events. Has optional internal activities: entry / (runs on entering), do / (runs while in state), exit / (runs on leaving).
  • Transition: labeled event [guard] / action. Event triggers it, guard is a boolean condition, action is an executable operation. All three parts are optional.
  • Composite state: a state that contains nested substates. Used to avoid repeating common transitions (e.g., a "Cancel" transition from any nested state).
  • Orthogonal regions: a composite state split into parallel regions (separated by a dashed line) — both regions are active simultaneously. Models objects with independent concurrent sub-behaviours.
  • Pseudostates: initial (●), fork, join, choice (◇), shallow history (H), deep history (H*).
  • Final states: activity final (◎) — ends the region; terminate pseudostate — destroys the object.

State diagrams are a direct specification for a finite state machine. They map to the State design pattern and can be mechanically converted to code (enum + switch, or the State pattern).

Order lifecycle state machine with composite state
Transition format: event [guard] / action

Example: paymentReceived() [amount == total] / sendReceipt()
Event: paymentReceived() — the trigger.
Guard: [amount == total] — only if true does the transition fire.
Action: /sendReceipt() — executed during the transition.
All three parts are optional. A transition without an event fires automatically (completion transition).

Use it when
  • Object has complex lifecycle — orders, subscriptions, user accounts, IoT device states, game entities.
  • Multiple boolean flags in your code signal a missing state machine. Replace isActive + isPending + isShipped with one state enum.
  • Network protocols — TCP connection states are a classic example.
  • UI component states — forms (idle, loading, error, success).
Don't use it when
  • Simple objects with 2 states — if it's just active/inactive, a boolean field plus a comment is enough.
  • Every class in the system — only model classes whose behaviour is significantly state-dependent.
  • Workflow across multiple objects — use an Activity Diagram with swimlanes instead.
Best Practices
  • One state machine per reactive object — don't put two objects' lifecycles on one diagram.
  • Use composite states to avoid repetition — if "Cancel" fires from PROCESSING, SHIPPED, and ALLOCATED, make those a composite FULFILLMENT state and add Cancel once.
  • Validate completeness — in each state, ask: "what happens if event X arrives unexpectedly?" Unhandled events should be shown explicitly (→ Error state or ignored with a note).
  • Implement with the State pattern or a state machine library — never use raw boolean flags. Spring Statemachine, XState (JS), or a simple enum + switch are all valid.
Common Mistakes
  • Conflating activity with state — a state is a condition of being ("PROCESSING"), not an action ("Validate"). If you're using gerunds (verbs ending in -ing), it's probably an activity diagram node, not a state.
  • Missing transitions for invalid events — "what happens if PAID receives a cancel() event?" must be answered: transition to CANCELLED, or mark as ignored. Gaps in the state machine map to undefined runtime behaviour.
  • Too many states — if you have more than 10–12 top-level states, use composite states or split the object into sub-objects. Real-life example: an Order doesn't need 20 states — group PROCESSING+PICKING+PACKING into one FULFILLMENT composite state.
  • Using state diagrams for workflow across objects — a State Machine models one object's lifecycle. If you're showing Customer AND Order AND Payment transitions together, use an Activity Diagram with swimlanes instead.

Sequence Diagram

The most used behavioral diagram. Shows time-ordered message exchanges between lifelines (objects/services). The go-to tool for API design and documenting complex call chains.

Theory

A Sequence Diagram has a time axis running downward. Lifelines are vertical dashed lines, each representing a participant. Activation boxes (thin rectangles on the lifeline) show when a participant is active (on the call stack).

Message types:

  • Synchronous (→ solid arrowhead): caller blocks until response. Like a regular method call.
  • Asynchronous (→ open arrowhead): caller continues immediately. Like an async event or fire-and-forget.
  • Return (⇢ dashed arrow): response from callee back to caller. Values are shown on the label.
  • Create (→ to lifeline box): message that creates a new object.
  • Destroy (✕ at end of lifeline): object is deleted.
  • Self-call (→ back to same lifeline): recursive call.

Combined Fragments (framed boxes with keywords) add conditional logic:

  • alt — if/else (one operand executes based on guards)
  • opt — optional (executes only if guard is true)
  • loop — repeats (with optional min/max count or condition)
  • par — parallel execution
  • break — abort and exit the enclosing fragment on condition
  • ref — reference to another Sequence Diagram
  • critical — atomic region (no interleaving)
Login flow with combined fragment (alt)
Use it when
  • API design — drawing the Sequence Diagram before coding ensures you think through all interactions and return values.
  • Documenting a complex feature that crosses multiple services or layers.
  • Code review — attach a sequence diagram to a PR to explain a non-obvious flow.
  • Debugging distributed systems — draw the actual flow from logs to find where things break.
Don't use it when
  • Showing every method call — diagrams with 40+ messages become unreadable. Stay at the design level, not implementation level.
  • Structural relationships — for "what owns what", use a Class Diagram.
  • Process flow with decisions — use an Activity Diagram when the flow branching is the main concern.
Best Practices
  • 10 lifelines maximum — beyond that, split into multiple diagrams linked with ref frames.
  • Name lifelines by role, not instance — :OrderService not orderServiceBean.
  • Show all return values — "fire and forget" vs "get result back" is a crucial design decision; make it explicit.
  • Use ref frames to delegate to other diagrams rather than nesting everything in one.
  • Synchronous vs async matters — a wrong arrow type implies wrong threading model.
Common Mistakes
  • Showing every internal method call — a sequence diagram that maps to the call stack 1:1 is a debugger output, not a design document. Stay at the collaboration level: services talking to services, not private helper methods calling private helper methods.
  • Too many lifelines — beyond 6–7 lifelines the horizontal space runs out and messages start overlapping. Split using ref frames to sub-diagrams.
  • Missing activation boxes — without them, you can't tell if a service is waiting (blocked) or returned. Every activate must have a matching deactivate.
  • Forgetting the return arrow — a call that has no return dashed arrow implies fire-and-forget. If the caller uses the result, draw the return.
  • Nesting alt blocks — deeply nested alt/opt/loop fragments create hard-to-read diagrams. If you have alt inside loop inside opt, split into sub-diagrams with ref.

Communication Diagram

Shows the same information as a Sequence Diagram, but emphasizes which objects collaborate rather than when. Messages are numbered to show sequence. Called "Collaboration Diagram" in UML 1.x.

Theory

A Communication Diagram is semantically equivalent to a Sequence Diagram — any sequence diagram can be converted to a communication diagram and vice versa. The difference is the emphasis:

  • Sequence Diagram: time is spatial (downward axis). Who calls who and in what order is visually obvious.
  • Communication Diagram: objects are arranged by structural relationship. Messages are numbered (1, 2, 3…) or hierarchically (1.1, 1.2, 1.1.1…) to show call order and nesting.

Numbering scheme: 1 = first top-level message. 1.1 = first nested call made during message 1. 1.2 = second call made during message 1. This Dewey-decimal numbering makes call depth visible without a Y-axis.

Links (solid lines between objects) represent the structural connections (associations, dependencies) that allow messages to flow. Links are permanent for the duration of the interaction.

Same login flow shown as a Communication Diagram
Use it when
  • Structural relationships matter more than timing — when you want to show which objects collaborate and through what links.
  • Design patterns — showing the Observer pattern's notification chain, or a Mediator coordinating multiple colleagues.
  • Limited drawing space — communication diagrams use less vertical space than sequence diagrams.
Don't use it when
  • Complex call ordering — Dewey numbering gets hard to follow beyond level 3 (1.1.1.2). Use a Sequence Diagram.
  • Real-time timing requirements — there's no time axis; use a Timing Diagram.
  • Parallel / concurrent flows — hard to represent; Sequence Diagrams with par fragments are clearer.
Best Practices
  • Keep numbering depth to 2 levels — if you need 1.1.1.2 you've exceeded what readers can track. Split into multiple diagrams.
  • Label links with what they represent structurally — write the interface or association name (: IOrderService) not just "has-a".
  • Use for design pattern documentation — the Observer, Mediator, and Strategy patterns are more intuitively documented as Communication Diagrams than Sequence Diagrams because the object graph structure is the insight.
  • Show return messages explicitly — a Communication Diagram without return arrows looks like everything is fire-and-forget. Add numbered dashed arrows for returns (e.g., 3.1: User entity).
  • Convert from Sequence Diagram as needed — if you have a Sequence Diagram for a complex flow, redrawing it as a Communication Diagram reveals which object owns the coordination responsibility.
Common Mistakes
  • No sequence numbers — without Dewey decimal numbers, a Communication Diagram is just an object graph. There is no way to determine execution order. Every message must be numbered.
  • Messages without structural links — arrows must travel along structural links that represent associations in the Class Diagram. Sending a message to an object you have no reference to is a design error — the diagram makes this impossible to hide.
  • No return messages — a diagram with only call messages looks like everything is fire-and-forget. Add numbered dashed return arrows (e.g., 1.1.1: : User entity) to show that callers receive responses.
  • Deep nesting — beyond 3 levels of Dewey decimal nesting (e.g., 1.1.1.2), a Sequence Diagram is always more readable. Switch notation when complexity exceeds what numbering can clearly express.

Interaction Overview Diagram

A hybrid: an Activity Diagram where each node can be an inline interaction or a reference (ref) to another diagram. Maps complex multi-flow scenarios as a "table of contents" for interactions. Added in UML 2.0.

Theory

The Interaction Overview Diagram answers "how do multiple interactions relate and in what order?" It uses the Activity Diagram's control flow notation (start, decision, fork, join, end) but replaces action nodes with:

  • Inline interaction (sd frame): a small embedded Sequence Diagram drawn directly in the node.
  • Interaction occurrence (ref frame): a reference box pointing to a separately named Sequence Diagram — like calling a subroutine.

Think of it as a control flow map across a family of interaction diagrams. It answers "which Sequence Diagrams execute, in what order, and under what conditions."

Interaction Overview is part of the "Interaction" family in UML 2.5, alongside Sequence, Communication, and Timing diagrams. It is the least-used diagram in the standard due to its complexity relative to its value in most projects.

Checkout journey — overview of all interactions
Use it when
  • A scenario spans many Sequence Diagrams — the Overview is their index and control flow.
  • Complex conditional and parallel interaction flows across multiple services that are too big for one diagram.
  • BPMN-style teams that need more behavioural detail than BPMN provides but want a high-level map.
Don't use it when
  • Scenarios fit in one Sequence Diagram — skip the overhead.
  • Teams unfamiliar with UML — this is one of the hardest diagrams to read without UML training.
  • Simple workflows — an Activity Diagram covers simple flows without the interaction complexity.
Best Practices
  • 5–7 interaction references maximum — more is a sign that the scenario itself needs splitting into two independent flows.
  • Match ref names exactly to Sequence Diagram titles — sd Authenticate User in the overview must be the exact title of the referenced Sequence Diagram. This creates a navigable document set.
  • Use inline sd frames only for short interactions (3–4 messages). Longer ones belong in their own Sequence Diagram referenced via ref.
  • Always show decision conditions as guards — a [auth OK?] decision without explicit guard labels forces readers to guess what the condition is.
  • Ask "is an Activity Diagram enough?" — if you don't need to show message-level detail, an Activity Diagram with swimlanes communicates the same flow to a broader audience with less complexity.
Common Mistakes
  • Inline details instead of ref frames — spelling out every message inside the overview defeats its purpose. Keep each interaction as a named ref box pointing to a separately-titled Sequence Diagram. The overview is a map, not a full territory.
  • Inconsistent ref names — the name inside the ref box must exactly match the title of the referenced Sequence Diagram (sd Authenticate User). A mismatch breaks the navigable document set and confuses readers who try to cross-reference.
  • No terminal node — every path through the overview must reach a stop node. An open branch implies the interaction has no defined end state, which is a formal error.
  • Using when Activity Diagram suffices — if none of the referenced interactions need message-level detail, an Activity Diagram with swimlanes covers the same flow and is readable by non-UML audiences.

Timing Diagram

Shows state changes of objects on a precise time axis — like a logic analyser or oscilloscope view. Critical for real-time systems, embedded firmware, network protocols, and SLA specifications. Added in UML 2.0.

Theory

A Timing Diagram shows how states change over time for one or more lifelines. The horizontal axis is time (can be concrete: ms, s; or relative). Each row (lifeline) has named states on the Y-axis and a waveform showing when transitions occur.

Two notation styles:

  • Waveform notation: the state is shown as a stepped waveform — the row occupies the "height" of the current state. Transitions are vertical edges. Used for multi-state objects (like network connection states).
  • Compact notation: state names are written inside horizontal bands. A vertical edge is a transition. Used when many states would make the waveform hard to read.

Key elements:

  • Timing constraint: {t1..t2} — the transition must happen within this time window.
  • Duration constraint: {d1..d2} — the object must remain in a state for at least d1 and at most d2.
  • Observation: a named time instant (tick mark on the time axis).
  • Message: a vertical (or diagonal) arrow between lifelines at a specific time instant.
  • Destruction: an X at the end of a lifeline to indicate the object is destroyed.
HTTP request lifecycle with SLA constraint
Use it when
  • Real-time systems — timing constraints (< 5ms response, must sample every 10ms) belong on a Timing Diagram.
  • Embedded firmware — interrupt handling, hardware signal timing, bus protocols (I2C, SPI).
  • SLA documentation — express performance requirements visually for ops teams.
  • Network protocol specification — TCP handshake, TLS negotiation timing.
Don't use it when
  • Business application development — almost never needed. A text note "response time < 200ms" is sufficient.
  • Many objects — beyond 4–5 lifelines, the diagram becomes a diagram of coloured spaghetti.
  • Non-time-critical scenarios — overkill. Use a Sequence Diagram.
Best Practices
  • Choose a concrete time scale — "ms" is more useful than a vague relative axis when there are real SLA requirements.
  • Add duration constraints explicitly — {d: 10..50ms} on a state band makes requirements testable.
  • Keep lifelines to 3–4 — more than that and the waveform becomes unreadable without a real oscilloscope tool.
  • Choose robust vs concise deliberately — robust (waveform) shows state duration visually; concise shows only transition points. Use robust for digital signals with discrete states, concise when you have many lifelines and care only about transitions.
  • Align vertical tick marks — when showing cause and effect across lifelines (e.g., request sent at t=30ms triggers server at t=31ms), draw a vertical marker at both points so causality is visually obvious.
Common Mistakes
  • Vague time axis — "early / late / after" labels are useless for SLA testing. Use concrete units (ms, µs, clock cycles) anchored to a defined reference point (t=0 = packet received, t=0 = system boot).
  • Missing duration constraints — a timing diagram without {d: min..max} annotations is a picture, not a specification. The constraints are what make the diagram testable: they can be extracted and turned into acceptance test assertions.
  • Mixing robust and concise without a legend — readers who don't know PlantUML or UML timing notation will not understand why some lifelines show boxes and others show lines. Add a brief legend or use one notation type consistently.
  • No state labels — every waveform segment must carry a readable state name (IDLE, PROCESSING, WAITING). A waveform with only up/down transitions and no labels is an oscilloscope trace, not a UML Timing Diagram.

Why other notations exist

BPMN 2.0 — Business Process Model and Notation

Standard: OMG BPMN 2.0 (2011), same body as UML. BPMN is the industry standard for business process documentation. It supersedes proprietary notations and integrates directly with process engines (Camunda, Activiti, Flowable, IBM BPM).

Key elements vs UML Activity Diagrams:

  • Events: BPMN has 9 event types (Start, End, Intermediate) × 10 triggers (Message, Timer, Error, Escalation, Signal, Compensation, Link, Terminate…). UML Activity has 2 (send signal, receive signal).
  • Gateways: Exclusive (XOR), Inclusive (OR), Parallel (AND), Event-Based, Complex — each with precise execution semantics. UML has Decision and Fork/Join.
  • Pools and Lanes: BPMN Pools = separate participants (different organisations). UML Swimlanes = actors within one process.
  • Collaboration Diagrams: BPMN models two pools exchanging messages = B2B integration. UML has no equivalent.
  • Executable BPMN: the XML behind a BPMN diagram IS the process engine configuration. The diagram IS the code. UML diagrams are documentation only.
UML Activity vs BPMN — when to use which

Use UML Activity when: the audience is software developers, the process involves algorithmic branching, the context is software design documentation.
Use BPMN when: business analysts are the audience, the process will be executed by a process engine (Camunda), the process crosses organisational boundaries, or you need event-driven processes.

C4 Model — Context, Container, Component, Code

Created by Simon Brown (2011, not an ISO standard but widely adopted). C4 is a zoom-in approach to architecture diagrams — four levels of abstraction, each for a different audience.

  • Level 1: System Context Diagram — your system in the centre, external users and external systems around it. Even non-technical executives can read this.
  • Level 2: Container Diagram — inside your system: web apps, APIs, databases, mobile apps, message queues. Each box is a separately deployable unit. Roughly equivalent to a UML Component Diagram (but simpler notation).
  • Level 3: Component Diagram — inside one container: Spring services, controllers, repositories. Roughly equivalent to a UML Package Diagram.
  • Level 4: Code Diagram — class/entity level. C4 says "use UML Class Diagrams here" — C4 explicitly delegates to UML for this level.

C4 uses a simplified box-and-arrow notation. It intentionally avoids the full UML notation to be accessible to all stakeholders. The Structurizr DSL is the canonical implementation — you define the model in code and it generates C4 diagrams automatically from the source.

C4 vs UML — the practical comparison

C4 Level 1 (Context) has no UML equivalent — closest is a Use Case Diagram without use cases.
C4 Level 2 (Container) ≈ UML Deployment + Component Diagram combined.
C4 Level 3 (Component) ≈ UML Package Diagram.
C4 Level 4 (Code) → use UML Class Diagram.

Many modern teams use C4 for architecture (levels 1–3) and UML for detail (State, Sequence, Class). This is the recommended combination.

ArchiMate 3.2 — Enterprise Architecture

Published by The Open Group (same body as TOGAF). ArchiMate models the full enterprise — strategy, business, application, technology, and physical layers — and the relationships between them.

Three core layers:

  • Business layer: business processes, actors, roles, products, business services.
  • Application layer: software applications, application services, data objects — this is where UML and ArchiMate overlap.
  • Technology layer: infrastructure — nodes, devices, networks, system software. Overlaps with UML Deployment Diagrams.

ArchiMate adds Strategy (capabilities, resources, value streams) and Physical (equipment, facilities, materials) layers in v3.0+. It also supports Motivation (goals, principles, requirements, constraints) and Migration & Implementation (work packages, deliverables).

ArchiMate is the tool when you need to trace from business strategy all the way down to server hardware in one coherent model — something UML was never designed to do.

Entity-Relationship Diagrams (ERD) — Crow's Foot Notation

Introduced by Peter Chen in 1976, standardized in various forms (IDEF1X, crow's foot, Chen notation, Barker). ERDs model relational database schemas — entities (tables), attributes (columns), and relationships with cardinality.

Notation variants:

  • Chen notation: rectangles = entities, diamonds = relationships, ellipses = attributes. Verbose but conceptually clear. Used in academia.
  • Crow's foot notation (Martin / Barker): lines end in crow's foot symbols for cardinality (many) and a single bar (one). Most popular in practice — used by Lucidchart, DBeaver, DataGrip, dbdiagram.io.
  • IDEF1X: DoD standard, strict formal notation. Used in government/defense data modelling.
  • UML Class Diagram can model the same content with multiplicity — many teams use Class Diagrams for database design because their UML tool already supports it. The ERD approach is often clearer for pure DB work because it explicitly shows column types.

ERDs focus on data persistence — what's stored and how. They don't model behaviour or processes. Pair them with Sequence Diagrams to show how data flows at runtime.

Data Flow Diagrams (DFD) — Gane-Sarson / DeMarco notation

Developed by Larry Constantine (1960s), popularized by Tom DeMarco (1979) and Gane-Sarson. DFDs model how data moves through a system — inputs, processes, outputs, and data stores.

Four elements: Processes (circles or rounded boxes) — transform data. Data stores (open rectangles / double-bar) — persist data. External entities (squares) — sources and sinks of data. Data flows (labeled arrows) — the data being transferred.

DFDs are hierarchical: Level 0 (Context Diagram) shows the whole system as one process; Level 1 explodes it into major sub-processes; Level N drills further. They are popular in security threat modelling (STRIDE methodology uses DFD-style context diagrams) and legacy structured analysis (pre-OO era).

DFD vs UML Activity vs Sequence

DFD: "What data flows where?" — focuses on data transformation, not control flow.
UML Activity: "What actions happen and in what order?" — focuses on workflow and control.
UML Sequence: "What messages are sent between objects?" — focuses on interaction protocol.
They model the same system from orthogonal perspectives. In modern teams, DFDs are mostly replaced by Activity + Sequence Diagrams for new projects, but DFDs remain popular in STRIDE security reviews.

SysML 2.0 — Systems Modeling Language

Published by OMG. SysML is a UML profile (and now its own language in v2.0) designed for systems engineering — hardware + software + people + processes — not just software OO design.

SysML 1.x reuses 7 of UML's 14 diagrams and adds 4 new diagram types:

  • Block Definition Diagram (BDD): like UML Class Diagram but for physical systems. A block can be a sensor, a motor, a software module, or a human team. Compartments: values (typed properties), parts (composition), references, ports, operations.
  • Internal Block Diagram (IBD): like UML Composite Structure Diagram — shows how blocks are wired internally with flow ports and connectors. Models signal and material flows.
  • Parametric Diagram: shows mathematical constraints between block properties — used for performance budgets (power, weight, timing deadlines).
  • Requirements Diagram: traces requirements to design elements with derive, satisfy, verify, refine, trace relationships. No UML equivalent.

SysML 2.0 (2024) is a clean-break rewrite with its own syntax (KerML + SysML 2 language), not backward compatible with SysML 1.x. It aims to be a proper systems modelling language with formal semantics, not just a UML extension.

Flowchart — ISO 5807 Standard Symbols

The oldest visual notation for algorithms and processes, standardized as ISO 5807 (1985, withdrawn but still universally used). Flowcharts predate UML by decades — they were the primary documentation tool for structured programming before OO arrived.

Standard ISO 5807 symbols:

  • Oval / Rounded rectangle: Terminator (Start / End) — marks entry and exit points.
  • Rectangle: Process — an action, step, or computation.
  • Diamond: Decision — a branching condition with Yes/No or True/False paths.
  • Parallelogram: Input/Output — data entering or leaving the process.
  • Cylinder / Open rectangle: Data store / database read-write.
  • Arrows: Control flow — direction of execution.

Flowcharts are appropriate for: algorithm documentation for non-technical audiences, interview prep (whiteboard coding), simple process documentation, and anywhere UML Activity's strict notation is overkill. Use UML Activity when you need swimlanes, fork/join semantics, or integration with OO design.

Mermaid — Text-to-Diagram for Developer Docs

Created by Knut Sveidqvist, open source (MIT). Mermaid renders diagrams from plain-text DSL inside Markdown — natively supported in GitHub, GitLab, Obsidian, Notion, Confluence, and VS Code. No tooling to install; the diagram lives as code next to your code.

Supported diagram types:

  • flowchart / graph: general-purpose flowcharts with TD, LR, BT, RL directions.
  • sequenceDiagram: UML-compatible sequence diagrams with alt/loop/par blocks.
  • classDiagram: UML class relationships — inheritance, composition, aggregation.
  • erDiagram: ERD with crow's foot cardinality and attribute lists.
  • stateDiagram-v2: state machines with transitions and nested states.
  • gantt: project timeline charts with sections and dependencies.
  • gitGraph: Git branching history visualization.
  • mindmap: radial mind map from indented text.
  • C4Context / C4Container / C4Component: C4 diagrams (v10.3+).

Mermaid is ideal for keeping diagrams in version control alongside code. The tradeoff: the DSL has layout limitations compared to hand-tuned diagrams. For precise visual control, use draw.io or PlantUML. For "diagram as code" developer experience, Mermaid is the first choice.

Network and Infrastructure Diagrams

There is no single ISO standard for network topology diagrams — the closest is Cisco's widely adopted icon set and notation, now available in draw.io, Lucidchart, and similar tools. Cloud providers (AWS, Azure, GCP) each publish official icon libraries for their services. The key distinction is between physical topology (what cables/hardware connect where) and logical topology (what IP addresses, subnets, and routing rules govern traffic).

  • Physical diagrams: racks, switches, routers, physical servers, data center layout. Used by network engineers and data center teams.
  • Logical diagrams: subnets, VPCs, security groups, DNS, routing tables. Used by DevOps, cloud architects, and security teams.
  • Cloud architecture diagrams: AWS/Azure/GCP service icons with arrows showing data flows and IAM boundaries. Used in cloud architecture reviews and documentation.
  • Deployment diagrams (UML): the formal UML equivalent — models deployment artifacts on nodes. Less specific than cloud-native notation for real infrastructure, but tooling-independent.

Best practice: use the official icon set from your cloud provider (download from their website). This ensures reviewers immediately recognize the services and apply the right mental model for capacity, latency, and cost assumptions.

Event Storming — DDD Collaborative Discovery

Created by Alberto Brandolini (2012). Event Storming is a workshop technique for rapidly exploring complex business domains using sticky notes on a large wall or virtual whiteboard. It is the primary discovery tool in Domain-Driven Design (DDD) for identifying bounded contexts, aggregates, commands, and domain events before designing a system.

Sticky note color conventions (Brandolini's palette):

  • Orange: Domain Events — facts that happened ("Order Placed", "Payment Failed"). Written in past tense. The core building block; start here.
  • Blue: Commands — what triggered the event ("Place Order", "Cancel Shipment"). Issued by actors or systems.
  • Yellow: Aggregates — the domain objects that process commands and emit events ("Order", "Cart", "Customer").
  • Purple: External Systems — third-party or external services that interact with the domain.
  • Pink / Red: Hotspots — unresolved questions, conflicts, or areas needing discussion.
  • Green: Read Models / Views — information displayed to the user that influences their next command.
  • Lilac: Policies / Business Rules — automatic reactions ("When payment fails, send dunning email").

Event Storming is not a formal notation standard — it is a facilitation practice. The output feeds directly into Bounded Context design and microservice boundary identification.

Wardley Maps — Strategic Technology Landscape

Created by Simon Wardley (2005). A Wardley Map is a strategic planning tool that visualizes the components of a system on two axes: the value chain (vertical — how visible is this to the end user?) and evolution (horizontal — how mature/commodity is this technology?). It was developed to address the lack of situational awareness in IT strategy discussions.

Two axes:

  • Y-axis (Value Chain): Visible to the user (top) → invisible infrastructure (bottom). Customer needs sit at the top; enabling components cascade down.
  • X-axis (Evolution): Genesis (novel, experimental) → Custom Built → Product → Commodity/Utility. All technology moves left-to-right over time.

Uses: identifying where to build vs buy vs use a managed service; spotting when a custom solution is about to become a commodity (disruption signal); communicating strategic decisions to executives without jargon. The map is a communication tool, not a formal specification — it should provoke debate.

Mind Maps — Radial Concept Mapping

Popularized by Tony Buzan (1960s). A mind map is a radial hierarchy diagram with a central concept and branches representing related ideas, sub-topics, and details. Used for brainstorming, note-taking, knowledge organization, and project planning at a conceptual level.

  • Mind map: radial, single-root, hierarchical. One idea leads to sub-ideas in a tree shape. Good for individual thinking and note-taking.
  • Concept map: more general graph — nodes can have multiple parents, labeled relationships. Good for modeling knowledge domains.
  • Mermaid mindmap: text-to-diagram mindmap using indented syntax, renders in GitHub/Obsidian. Practical for developer documentation.
  • XMind / MindMeister / Obsidian: dedicated tools with presentation modes and export to various formats.

Mind maps are informal — no standard body governs them. They are best as thinking aids and early-stage planning tools, not as precise technical specifications. For structured requirements, switch to a Requirements Diagram (SysML) or backlog format.

Gantt Charts — Project Scheduling and Roadmaps

Developed by Henry Gantt (1910–1915), refined by Karol Adamiecki. Gantt charts model project schedules — tasks on the Y-axis, time on the X-axis, with horizontal bars showing duration, dependencies shown as arrows, and milestones as diamonds. ISO 8601 governs date/time formats used in them.

  • Classic Gantt (Visio / MS Project): task list on left, timeline grid on right, dependency arrows, resource assignments, critical path highlighted.
  • Mermaid gantt: text-based gantt blocks rendering in Markdown. Supports sections, tasks, crit/active/done status, date expressions.
  • Roadmaps: high-level Gantt for product strategy — quarters instead of days, features instead of tasks. Used in product management (Jira Roadmaps, Aha!, Notion timeline).
  • Critical path method (CPM): mathematical extension of Gantt — computes which tasks, if delayed, will delay the whole project. Built into MS Project and Primavera.

For developer documentation, Mermaid's gantt block is the most practical option — version-controlled alongside code. For project management, dedicated PM tools (Jira, Asana, Linear) are better because they link tasks to actual tickets and resource assignments.

Wireframes and Mockups — UI Design Notation

Wireframes and mockups are UI design artifacts, not a formal notation standard. They communicate layout, information hierarchy, and interaction flow without committing to visual design decisions (colors, fonts, imagery). The terms form a spectrum:

  • Sketch / napkin drawing: hand-drawn boxes and arrows. Fastest; used in early ideation.
  • Low-fidelity wireframe: grayscale, placeholder content (Lorem Ipsum, grey boxes for images), no visual design. Purpose: agree on structure. Tools: Balsamiq (deliberately sketch-like), draw.io, Whimsical.
  • High-fidelity mockup: pixel-precise design with real colors, fonts, and content. Looks like the final product. Purpose: sign-off before development. Tools: Figma, Sketch, Adobe XD.
  • Interactive prototype: clickable high-fi mockup that simulates user flow. Purpose: user testing without building real code. Tools: Figma prototyping, InVision, Axure.

Wireframes should come before both visual design and implementation. Skipping wireframes and jumping to code is a frequent cause of rework — stakeholders can't give structured feedback on code, but they can annotate a wireframe. Use UML Activity Diagrams alongside wireframes to document the user workflow; use Sequence Diagrams for the API calls each action triggers.

1 · Class Diagram — The Domain Model

First thing any architect draws: what things exist in this system and how they relate.

Why Class Diagram first? It is the backbone of the entire project. Every other diagram references these concepts. A composition diamond on ParkingLot→ParkingSpot tells your team that spots are destroyed when a lot is deleted — a data-integrity rule that belongs in the model, not buried in a README. The 1:1 Sensor→Spot relationship immediately surfaces a hardware coupling decision that affects both schema design and deployment.
Key insight: Notice Reservation is purple (behavioral) while ParkingSpot is blue (structural). That split reveals that a Spot is a long-lived entity, but a Reservation is a transient process artifact — a subtle but critical distinction for your cache-invalidation strategy.
2 · Object Diagram — A Snapshot in Time

Class diagrams are abstract. Object diagrams answer: "what does slot A5 look like at 09:15 on a Tuesday?"

Why Object Diagram? When your team debates "should a cancelled reservation still link to a payment?", an object diagram makes the answer visible instantly. sensor_A5 with batteryLevel = 87% immediately prompts the question: should we alert the admin below 20%? You would never ask that question from the class diagram alone.
Key insight: Object diagrams are the best bug reports. When you file "A5 shows available but a car is there", this exact object diagram — with isAvailable = false but status = ACTIVE — is the data state you hand to the developer. Far better than plain text.
3 · Component Diagram — Microservices Architecture

Now we zoom out: how do the software modules fit together?

Why Component Diagram? This is the first diagram you show to a new backend developer. In 30 seconds they understand which service owns which responsibility, that IoT sensors never call the Reservation Service directly (they go through Sensor Hub), and that Payment Service is a leaf — nothing calls it except Reservation Service. That last point means Payment Service can be scaled independently without touching anything else.
Key insight: The dashed line from API Gateway → Payment Service is intentional: mobile clients should never call Payment directly. That architectural rule lives here, enforced in code reviews by pointing at this diagram. Without it, someone will add a shortcut "just this once."
4 · Deployment Diagram — Where Does It Actually Run?

Components tell you what runs. Deployment tells you where — the physical and cloud reality.

Why Deployment Diagram? Your DevOps engineer, cloud architect, and security auditor all need different things from the same system — and this one diagram serves all three. The PCI-DSS zone around Payment Service is invisible in every other diagram but legally required in this one. "Payment Service should be in its own VPC subnet" is a deployment concern, not a design concern — this is the only diagram that captures it.
Key insight: Redis (ElastiCache) does not appear anywhere in the Class or Component diagrams — it is a deployment-level optimisation. Spot availability reads hit Redis first; writes go to Aurora. That caching contract lives here. If you leave it out, the team re-debates it every sprint.
5 · Package Diagram — Code Organisation

Components show runtime modules. Package diagrams show the source-code folder structure and the compile-time dependency rules.

Why Package Diagram? This enforces the Dependency Inversion Principle at the folder level. The domain package has zero outgoing arrows — it knows nothing about databases or HTTP. That single rule, made visible here, prevents the most common enterprise architecture smell: business logic that imports pg.Pool directly. Your CI lint rule references this diagram: "no import from infra in domain".
Key insight: The infra package points at domain (not services). This is the Repository Pattern: PostgresRepository implements a domain interface. Switch from Postgres to MongoDB? Only the infra package changes. Show a junior developer this arrow and they instantly understand why the swap is safe.
6 · Use Case Diagram — What the System Does for Whom

Before writing a single endpoint, align with stakeholders on the scope of the system.

Why Use Case Diagram? The first use-case diagram you draw should fit on a whiteboard and survive a 10-minute stakeholder meeting. Notice that Admin cannot Reserve or Pay — this is a business rule encoded in the diagram. The «include» from Reserve → Pay means payment is always part of booking (not optional), while «extend» on Extend Reservation → Pay means extension might trigger a charge — a subtle but important difference.
Key insight: Use case diagrams are the only UML diagrams your product manager will actually read. Print this, get sign-off, and it becomes the acceptance criteria baseline. Every sprint, point to it: "Did we ship all the Driver use cases? Yes? Sprint done." Deceptively simple, irreplaceable in practice.
7 · Activity Diagram — The Booking Flow

How does the system actually execute "Reserve a spot"? Activities answer that — including who does what.

Why Activity Diagram? Swim lanes reveal a classic race condition: two drivers select the same spot simultaneously. The "Spot still free?" decision diamond after spot selection (not before) catches this — the spot must be checked AND locked atomically. Without drawing this flow, that bug ships to production. It was caught here, in a diagram review, before a line of code was written.
Key insight: The fork after "Lock spot" reveals that reservation creation and payment charging can run in parallel. That is a performance optimisation born from drawing the diagram — if both succeed, great; if payment fails, the reservation rolls back. Without the fork notation, developers would naturally implement them sequentially and never question it.
8 · State Machine — The ParkingSpot Lifecycle

A parking spot does not just toggle on/off. It has a lifecycle — and every transition has a rule.

Why State Machine? Before this diagram, the team was implementing availability as a boolean isAvailable. The state machine reveals four distinct states with completely different behaviour. MAINTENANCE alone changes the database schema: you now need a maintenanceReason column and an admin notification hook. That design change is free at diagram stage, expensive after shipping.
Key insight: The dashed "timerExpired" self-transition on RESERVED is the most valuable line in this diagram. A reservation with no timeout means a driver can book a spot and never show up, blocking it forever. This is a product decision (10-minute grace period?) that looks like a code bug without the diagram — and a legal liability in pay-per-minute systems.
9 · Sequence Diagram — The Booking Call Stack

Activity diagrams show what happens. Sequence diagrams show who calls whom — the exact message ordering across objects.

Why Sequence Diagram? The "alt" fragment for payment failure is the moment this diagram earns its keep. Message 3.1 locks the spot, then 3.2 charges the card. If 3.2 fails, someone must call unlockSpot() — and without this diagram nobody noticed that responsibility falls on Reservation Service, not the caller. This is a compensation pattern that must be explicit; it does not emerge from a sprint ticket.
Key insight: The activation boxes show that API Gateway stays active for the entire call chain. If Reservation Service is slow, the driver's HTTP connection times out at the gateway — which has a hard 29-second AWS limit. Drawing the boxes makes this latency budget visible. The team immediately decided to make step 3.2 async (return reservation ID first, confirm payment via push notification) to beat that timeout.
10 · Communication Diagram — Object Collaboration Network

Same scenario as the Sequence Diagram but drawn differently — emphasising which objects talk to which, not the order.

Why Communication Diagram (alongside Sequence)? The dashed "never direct" line between DriverApp and ParkingService is invisible in a Sequence Diagram — you just don't draw the arrow. Here it is explicitly called out as a design constraint. Also notice: Notification Service has exactly one incoming arrow. That is a fanout boundary — if you need Slack alerts tomorrow, you add it inside NotificationService, nowhere else.
Key insight: A Sequence Diagram answers "what order?" A Communication Diagram answers "how many connections does each object have?" PaymentService has exactly one caller. That is the architectural invariant you want to enforce — and it is much more obvious from this diagram than from the sequence. Use both together; they are the same information from two angles.
11 · Timing Diagram — Real-Time Constraints

ParkSmart has a hardware backbone. Sensors must update spot status fast enough that drivers don't navigate to an already-taken spot.

Why Timing Diagram? This is the only diagram that reveals a silent UX failure: there is a ~1.5 second window where the sensor says "car arrived" but the mobile app still shows "FREE". If a second driver books during that window, they get a false confirmation. The timing diagram makes this race condition measurable — it becomes an SLA: sensor-to-UI latency must be ≤ 1.5 seconds. Your SRE team now has a Prometheus alert threshold.
Key insight: No other UML diagram can express "this must happen in under 1 second." Timing diagrams are rare in web applications but essential in IoT, embedded, and real-time systems. The moment you add hardware to your stack, reach for this diagram first. It turns vague requirements ("it should be fast") into hard constraints with enforcement teeth.
12 · Composite Structure — Inside the ReservationEngine

A Component Diagram shows you the module exists. A Composite Structure Diagram shows you what is wired together inside it — parts, ports, and connectors.

Why Composite Structure? The Component Diagram tells you ReservationEngine exists. This diagram tells you what is inside it and how those parts talk to each other. The port-based boundary (IBook, INotify, IPay) means external callers never reference AvailabilityChecker or SpotLockManager directly — they can be replaced without changing the outer contract. The connector from SpotLockManager → PaymentCoordinator makes the ordering dependency explicit: you cannot charge the card before the spot is locked.
Key insight: The saga pattern inside PaymentCoordinator (compensation on failure) is a structural decision, not just a runtime behaviour. Drawing it as a part with a clear rollback responsibility means developers know which class owns the undo logic. Without this diagram, compensation code ends up scattered across three services and nobody owns the failure path.
13 · Profile Diagram — Domain-Specific UML Extensions

Standard UML has no concept of "IoT sensor" or "PCI-scoped class." A Profile Diagram lets you extend UML itself with your own stereotypes and tagged values.

Why Profile Diagram? Every IoT sensor in ParkSmart has a pollingInterval, batteryLevel, and protocol. Without a profile, these tagged values live in a Word doc nobody reads. With a profile, they are first-class UML elements — a code generator reads the «IoTSensor» stereotype and auto-generates the MQTT connection boilerplate. The «PCIScope» stereotype does the same for payment classes: any class marked with it automatically gets encryption-at-rest wiring in the generated output. Documentation becomes executable.
Key insight: Profile Diagrams are the most underused diagram in the UML specification. They shine in two situations: large teams where consistent metadata must be enforced across hundreds of classes, and code-generation pipelines where model annotations drive output. In ParkSmart, the «PCIScope» stereotype is the single source of truth for PCI compliance — a compliance auditor can count the stereotype usages instead of reading every class file.
14 · Interaction Overview — The End-to-End User Journey

When the full journey involves six Sequence Diagrams, an Interaction Overview knits them into one readable map.

Why Interaction Overview? The ParkSmart journey has three separate Sequence Diagrams (discovery, booking, navigation). Showing all three in one IOD lets a product manager, a QA engineer, and a backend developer each find their piece without reading the other two. The two decision diamonds surface the only two unhappy paths in the entire system from a user's perspective — making test plan coverage trivial to derive.
Key insight: The IOD makes it immediately obvious that there are two early exits in the user journey — no spot available, and payment failure. Both lead to dead-end final nodes. That means no retry loop is built into the happy path, which is a product decision that should be explicit. This diagram surfaced it in a 5-minute review; without it, the retry logic appeared three weeks later as a last-minute feature request.
